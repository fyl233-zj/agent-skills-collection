> 适配工具：Claude Code / Cursor / Codex / OpenCode / Hermes Agent

## 使用说明

每个 Skill 包含四个部分：

| 部分 | 说明 |
|------|------|
| 触发条件 | 说什么话、做什么操作会激活这个 Skill |
| 执行步骤 | Agent 收到触发后的具体行为流程 |
| 适用场景 | 什么情况下该用这个 Skill |
| 常见坑 | 中文开发者使用时最容易踩的坑 |

**安装方式**：将 Skill 内容保存为 .md 文件放入 Agent 的 skills 目录即可。

## 第一章：代码质量（4个）

### Skill 1：代码审查 Code Review

**触发条件**：用户说"review""审查""检查这个 PR""看看代码"

**执行步骤**：
1. 读取所有变更文件的 diff
2. 逐文件按四维检查：
   - 安全漏洞：SQL 注入、XSS、CSRF、密钥硬编码、敏感信息日志输出
   - 边界条件：null/undefined、数组越界、除零、空字符串
   - 性能瓶颈：N+1 查询、循环内 IO、不必要的深拷贝、内存泄漏模式
   - 规范一致性：命名风格、文件结构、错误处理模式
3. 输出三级分类报告：
   - 🔴 阻断性：必须修改才能合入
   - 🟡 建议性：建议修改，给出理由
   - 🔵 风格性：可选，统一风格
4. 每个问题附带：所在文件+行号 + 具体修改建议代码片段

**适用场景**：PR Review、代码走查、上线前质量检查

**常见坑**：
- 不要让它审查超过 500 行变更——大 PR 拆成小 PR 再审查
- 安全审查需要额外告知项目使用的框架（Express/Django/Spring 等），否则只会做通用检查
- 审查结果不等于人工审查，涉及业务逻辑的地方需要人工确认

---

### Skill 2：Bug 诊断 Systematic Debug

**触发条件**：用户说"报错了""不工作""有 bug""出问题了"

**执行步骤**：
1. **禁止直接改代码**。先执行以下诊断：
2. 读取完整错误日志和堆栈信息
3. 确认三要素：预期行为、实际行为、触发条件
4. 如果信息不足，在关键路径加中间值打印（不修改业务逻辑）
5. 构造最小复现用例
6. 确认根因后，输出诊断报告（根因+证据+修复方案）
7. 用户确认后再实施修复

**适用场景**：任何报错、异常行为、测试失败

**常见坑**：
- 国内开发者习惯直接改代码，这个 Skill 强制先诊断——要忍住不改
- Python 项目的编码问题（GBK/UTF-8）是高频根因，但 Agent 不会主动怀疑编码
- 异步代码的 bug 通常需要加时序日志而非静态分析

---

### Skill 3：测试生成 Test Generation

**触发条件**：用户说"写测试""加测试用例""test"

**执行步骤**：
1. 读取目标函数/模块的完整代码
2. 提取：输入参数类型、返回值类型、所有分支条件、异常抛出点
3. 按优先级生成测试用例：
   - P0：正常路径（happy path）
   - P1：边界条件（空值、极值、零值）
   - P2：异常路径（错误输入、超时、网络异常）
   - P3：并发/竞态（如有必要）
4. 每个用例包含：Given-When-Then 三段式 + 断言
5. 自动适配项目的测试框架（pytest/Jest/JUnit 等）

**适用场景**：新功能开发后补测试、重构前补测试、提高覆盖率

**常见坑**：
- 前端测试需要明确测试框架（Jest/Vitest/Cypress），否则生成不兼容
- 中国项目的 API 测试需要指明 Mock 方式——国内常用 Mock.js 而非 Sinon
- 数据库相关测试必须生成事务回滚逻辑，否则污染测试库

---

### Skill 4：代码重构 Code Refactoring

**触发条件**：用户说"重构""优化这段代码""写得太烂了"

**执行步骤**：
1. 分析目标代码的 4 个维度：
   - 复杂度：圈复杂度、嵌套深度、函数长度
   - 重复度：相似代码块、重复逻辑
   - 可读性：命名、注释、结构
   - 可测试性：依赖关系、副作用
2. 给出重构方案（不改代码，先出方案）：
   - 要拆哪几个函数
   - 要改哪几个命名
   - 要抽象哪几个公共逻辑
3. 用户确认方案后实施
4. 重构后自动运行已有测试，确保行为不变

**适用场景**：老代码优化、接手别人的代码、技术债务清理

**常见坑**：
- 重构前必须确保已有测试覆盖——否则改完不知道有没有破坏功能
- Python 项目注意不要引入循环导入
- 前端重构注意不要改动组件的外部接口（props/events），否则调用方全崩

## 第二章：文档（3个）

### Skill 5：API 文档生成

**触发条件**：用户选中函数/模块说"写文档""生成 API 文档"

**执行步骤**：
1. 解析函数签名：参数名、类型注解、默认值、返回值类型
2. 识别所有异常路径和边界条件
3. 生成结构化文档：
   - 功能描述（一句话）
   - 参数表（名称 | 类型 | 必填 | 默认值 | 说明）
   - 返回值（类型 + 说明）
   - 异常（类型 + 触发条件）
   - 3 个示例（正常 + 边界 + 异常处理）
   - 注意事项

**适用场景**：接口文档、SDK 文档、内部库文档

**常见坑**：
- 类型注解不完整的 Python 函数，Agent 会猜错类型——先补注解
- 国内项目常混合中英文注释，告诉 Agent 输出语言偏好
- 不要让它生成"所有函数的文档"——一次一个模块，质量更高

---

### Skill 6：README 生成

**触发条件**：用户说"写 README""生成项目说明"

**执行步骤**：
1. 扫描项目结构：package.json/setup.py/Cargo.toml、入口文件、目录结构
2. 提取：项目名、依赖、入口、脚本命令
3. 生成标准 README：
   - 项目名 + 一句话描述
   - 快速开始（安装 + 运行）
   - 功能特性
   - 项目结构
   - 配置说明
   - 贡献指南
4. 自动添加 Badge（构建状态、版本、License）

**适用场景**：开源项目、内部工具文档化

**常见坑**：
- 中国项目通常 README 中英文混合——告诉 Agent 语言策略
- 安装步骤要适配国内镜像源（npm/pip 镜像），否则读者装不上
- Badge 链接可能指向被墙服务（如 shields.io 某些资源），建议用国内替代

---

### Skill 7：更新日志生成 Changelog

**触发条件**：用户说"写 changelog""更新日志""发版说明"

**执行步骤**：
1. 读取 git log（从上一个 tag 到 HEAD）
2. 按约定式提交分类：feat/fix/docs/refactor/perf/test/chore
3. 生成结构化的 changelog：
   - 版本号 + 日期
   - 新增功能（feat）
   - Bug 修复（fix）
   - 文档更新（docs）
   - 性能优化（perf）
   - 破坏性变更（BREAKING CHANGE）
4. 每条附上 commit 摘要

**适用场景**：发版时、GitHub Release 页面

**常见坑**：
- 前提是团队用约定式提交——如果 commit 信息乱写，先规范提交习惯
- 中文 commit 信息需要告诉 Agent 保留原文还是翻译
- 破坏性变更的判断需要人工复核——Agent 不总能识别 API 变更的影响范围

## 第三章：安全（2个）

### Skill 8：安全审计 Security Audit

**触发条件**：用户说"安全检查""安全审计""看看有没有漏洞"

**执行步骤**：
1. 识别项目技术栈（框架、语言、数据库）
2. 按 OWASP Top 10 逐项检查：
   - 注入（SQL/NoSQL/OS 命令）
   - 认证失效（弱密码策略、Session 管理）
   - 敏感数据暴露（日志中的密码、响应中的内部信息）
   - XML 外部实体（XXE）
   - 访问控制失效
   - 安全配置错误
   - XSS
   - 不安全的反序列化
   - 使用含已知漏洞的组件
   - 日志和监控不足
3. 输出：漏洞级别 + 位置 + 攻击向量 + 修复代码

**适用场景**：上线前检查、合规审计、第三方依赖审查

**常见坑**：
- Agent 只做静态检查，不会做渗透测试——不能替代专业安全审计
- 需要显式告诉 Agent 项目的部署架构（CDN/WAF/反向代理），否则会漏掉架构层面的漏洞
- 密钥泄露检查只扫硬编码，不会扫 .env 文件是否误提交——需要单独检查 git 历史

---

### Skill 9：依赖漏洞检查

**触发条件**：用户说"检查依赖""依赖安全""第三方包有没有问题"

**执行步骤**：
1. 读取依赖文件（package.json/requirements.txt/Cargo.toml 等）
2. 逐依赖检查：版本号、已知 CVE、维护状态（最后更新时间）
3. 标记风险级别：
   - 🔴 有已知严重漏洞 + 已停止维护
   - 🟡 有已知中危漏洞 或 超过 1 年未更新
   - 🟢 安全且活跃维护
4. 对每个 🔴 依赖给出替代方案

**适用场景**：项目初始化、定期安全巡检

**常见坑**：
- Agent 自身的漏洞数据库不是实时的——重大漏洞建议去 CVE 官网确认
- 国内项目常用的一些镜像源里的包可能版本滞后
- 间接依赖（依赖的依赖）也需要检查——Agent 默认只查直接依赖

## 第四章：性能（2个）

### Skill 10：性能分析 Performance Profiling

**触发条件**：用户说"太慢了""性能优化""profiling"

**执行步骤**：
1. 识别性能瓶颈类型：CPU 密集 / IO 密集 / 内存 / 网络
2. 定位热点代码：
   - 前端：检查渲染次数、bundle 大小、网络请求瀑布
   - 后端：检查数据库查询次数、循环复杂度、缓存命中率
3. 输出优化建议优先级：
   - P0：改一行代码见效（如添加索引、消除 N+1）
   - P1：改一个函数见效（如用缓存、批量查询）
   - P2：需要架构调整（如异步化、分库分表）
4. 每个优化附带：当前耗时 → 预期耗时 + 实现代码

**适用场景**：接口慢、页面卡、内存泄漏

**常见坑**：
- Agent 只能做静态分析——实际性能数据需要你跑压测后提供
- 优化前端性能时先看 Network 面板的瀑布图，Agent 猜不到你的网络环境
- Python 性能问题先怀疑 GIL 和 IO 阻塞——Agent 默认按通用模式分析

---

### Skill 11：数据库查询优化

**触发条件**：用户说"查询慢""优化 SQL""数据库性能"

**执行步骤**：
1. 读取 SQL 语句或 ORM 查询代码
2. 分析：
   - 是否命中索引（检查 WHERE/JOIN/ORDER BY 字段）
   - 是否存在 N+1 查询（ORM 特有）
   - 是否返回不必要字段（SELECT * 问题）
   - 是否可以用批量查询替代循环查询
   - 是否有慢查询模式（LIKE '%xxx%'、大表 JOIN、子查询嵌套）
3. 输出优化方案 + 改写后的代码/SQL

**适用场景**：列表接口慢、报表查询超时、数据库 CPU 飙升

**常见坑**：
- 没有执行计划的优化是盲目的——要求 Agent 先建议 EXPLAIN 再看结果
- ORM 的 N+1 问题 Agent 能发现但修复方案可能破坏业务逻辑——需要人工确认
- 中国项目用 MyBatis 多，Agent 默认给 SQLAlchemy/Django ORM 方案——需要明确技术栈

## 第五章：工程实践（4个）

### Skill 12：Git 工作流 Git Workflow

**触发条件**：用户说"提交代码""发 PR""合并""解决冲突"

**执行步骤**：
1. 检查当前分支状态（未提交变更、未推送提交、与主分支差异）
2. 按约定式提交规范生成 commit message：
   - feat: 新功能描述
   - fix: 修复描述
   - refactor: 重构描述
3. 如果有冲突，分析冲突原因并给出合并策略
4. 生成 PR 描述：变更概要 + 测试说明 + 截图（前端变更）

**适用场景**：日常提交、发 PR、处理合并冲突

**常见坑**：
- 不要让它 force push——Agent 对 git 历史的理解有限
- 冲突解决策略需要人工确认——Agent 不了解业务意图
- 中文 commit message 需要在团队内统一规范（纯中文/纯英文/混合）

---

### Skill 13：CI/CD 配置

**触发条件**：用户说"配置 CI""加流水线""GitHub Actions""Jenkins"

**执行步骤**：
1. 识别项目技术栈和部署目标
2. 生成 CI 配置文件，包含：
   - 代码检出
   - 依赖安装（使用国内镜像）
   - 代码检查（lint + type check）
   - 测试运行
   - 构建打包
   - 部署（可选）
3. 每个步骤附带超时设置和失败处理策略

**适用场景**：新项目初始化、CI 迁移、流水线优化

**常见坑**：
- 国内 CI（如 Gitee Go、CODING）的语法和 GitHub Actions 不同——明确平台
- npm/pip 安装步骤必须配置国内镜像，否则 CI 大概率超时
- 需要告知部署目标（阿里云/腾讯云/自建服务器），不同目标配置差异大

---

### Skill 14：错误处理标准化

**触发条件**：用户说"统一错误处理""异常规范""错误码"

**执行步骤**：
1. 扫描项目中所有 try-catch 和错误处理逻辑
2. 评估当前错误处理的一致性问题：
   - 是否统一了错误码
   - 是否区分了业务异常和系统异常
   - 是否有吞异常（空 catch）的情况
   - 错误信息是否暴露了内部细节
3. 生成标准化方案：错误码枚举 + 统一异常类 + 全局异常处理中间件
4. 给出迁移步骤（新旧并存过渡）

**适用场景**：后端项目规范化、微服务错误传递、前端统一错误提示

**常见坑**：
- 错误码设计需要考虑团队习惯——Agent 给的错误码规范可能和现有系统不一致
- 不要一次性改全部错误处理——先定规范，新代码用新规范，旧代码渐进迁移
- 前端错误处理要注意区分"用户看得到的提示"和"开发调试用的日志"

---

### Skill 15：代码迁移 Code Migration

**触发条件**：用户说"升级版本""迁移到""从 A 换到 B"

**执行步骤**：
1. 分析源技术栈和目标技术栈的差异
2. 识别迁移范围：API 变更、配置变更、语法差异
3. 生成迁移清单（按文件列出需要改什么）
4. 逐文件迁移，每个文件迁移后说明改了什么、为什么这么改
5. 生成迁移前后对比文档

**适用场景**：框架升级、语言版本升级、数据库迁移

**常见坑**：
- 大项目不要一次全迁——先迁一个模块验证
- 迁移过程中可能引入新 bug，每迁一个模块跑一次测试
- Python 2→3 / Vue 2→3 这类大版本迁移，Agent 覆盖不全，需要人工补充

## 第六章：架构与设计（2个）

### Skill 16：架构审查 Architecture Review

**触发条件**：用户说"看看架构""架构有问题吗""设计评审"

**执行步骤**：
1. 分析项目的目录结构、模块依赖关系、数据流
2. 从 5 个维度评估：
   - 模块划分：是否高内聚低耦合
   - 依赖方向：是否有循环依赖、依赖倒置
   - 数据一致性：是否有分布式事务风险、数据冗余
   - 扩展性：新增功能是否需要改多处代码
   - 复杂度：是否有过度设计、不必要的抽象层
3. 输出每个维度的评分和具体改进建议

**适用场景**：新项目设计评审、老项目重构前评估

**常见坑**：
- Agent 只能分析代码层面的架构——部署架构（微服务拓扑、数据库主从）需要你提供架构图
- 不要让它改架构——只做评估，改架构是人工决策
- 中国互联网项目常见的架构模式（读写分离、分库分表、CDN 加速）Agent 能识别但不会主动评估

---

### Skill 17：组件设计 Component Design

**触发条件**：用户说"设计一个组件""封装一个模块"

**执行步骤**：
1. 明确组件的输入（props/参数）、输出（events/返回值）、副作用
2. 设计接口优先——先定义对外接口，再实现内部逻辑
3. 考虑：可复用性、可测试性、可扩展性
4. 生成组件代码 + 使用示例 + 单元测试骨架

**适用场景**：前端组件封装、后端公共模块抽取、SDK 设计

**常见坑**：
- 前端组件不要一次设计太复杂——先满足当前需求，预留扩展点而非提前实现
- 后端模块设计要遵循"依赖倒置"——高层模块不依赖低层模块，都依赖抽象
- 命名是组件设计最难的部分——让 Agent 提供 3 个备选命名

## 第七章：工具与环境（3个）

### Skill 18：Docker 配置

**触发条件**：用户说"写 Dockerfile""容器化""docker-compose"

**执行步骤**：
1. 分析项目依赖：运行时、系统库、环境变量、端口、数据卷
2. 生成多阶段 Dockerfile：
   - 构建阶段（安装依赖、编译）
   - 运行阶段（最小化镜像、非 root 用户）
3. 如有多服务，生成 docker-compose.yml（含网络、数据卷、健康检查）
4. 附 .dockerignore 文件

**适用场景**：项目容器化、统一开发环境、微服务部署

**常见坑**：
- 国内镜像源（apt/pip/npm）必须在 Dockerfile 中配置，否则构建超时
- 不要用 latest 标签——固定版本号
- Node.js 项目需要 .dockerignore 排除 node_modules，否则构建极慢

### Skill 19：环境搭建 Environment Setup

**触发条件**：用户说"搭环境""跑不起来""环境配置"

**执行步骤**：
1. 读取项目的依赖文件和 README
2. 检查当前环境的：Node/Python/Java 版本、已安装的全局工具
3. 逐步骤验证安装过程：
   - 版本兼容性检查
   - 依赖安装（使用国内镜像）
   - 配置文件生成（.env 模板）
   - 数据库初始化
   - 启动验证
4. 每步失败时给出具体错误原因和修复方法

**适用场景**：新人入职、换电脑、项目交接

**常见坑**：
- Python 项目先检查系统 Python 版本和虚拟环境——国内很多机器的 Python 版本是 2.x+3.x 混合
- Node 项目先检查 node-gyp 和原生模块——Windows 上原生模块编译是最常见的环境问题
- 数据库初始化前必须检查端口冲突——国内机器常同时装了 MySQL 5.x 和 8.x

### Skill 20：日志分析 Log Analysis

**触发条件**：用户说"看看日志""分析日志""日志报了什么错"

**执行步骤**：
1. 读取日志文件或日志流
2. 按优先级处理：
   - 先过滤 ERROR/FATAL
   - 再过滤 WARN（可能无关，但需要关注）
   - 最后按时间线串联 INFO 重建请求链路
3. 输出：
   - 错误摘要（什么错误、几次、首次/末次时间）
   - 根因推测（基于日志上下文）
   - 建议排查方向
4. 如果日志量过大，自动聚合相似错误

**适用场景**：线上问题排查、压测结果分析、定时任务失败诊断

**常见坑**：
- 中文编码日志（GBK）需要先转码——Agent 默认按 UTF-8 读取会乱码
- 分布式系统的日志需要按 traceId 串联——单纯按时间排序看不到完整链路
- 不要让它分析超过 100MB 的日志文件——先 grep 关键词缩小范围

### 如何安装这些 Skills

**Claude Code**：将 Skill 的 .md 文件放入 `~/.claude/skills/` 目录

**Cursor**：将 Skill 内容复制到 `.cursorrules` 文件中

**Codex CLI**：将 Skill 的 .md 文件放入 `~/.codex/skills/` 目录

**Hermes Agent**：使用 `hermes skill create` 命令或放入 skills 目录

### 如何编写自己的 Skill

1. 确定一个具体场景（不要试图做一个"万能 Skill"）
2. 写清楚触发条件（什么话、什么操作会触发）
3. 写清楚执行步骤（像写 SOP 一样精确）
4. 测试：故意触发它，看 Agent 是否按预期执行
5. 迭代：发现漏掉的情况就补充

### 常见失败模式

| 问题 | 原因 | 解决 |
|------|------|------|
| Agent 不触发 Skill | 触发条件写得太模糊 | 用具体的关键词 |
| Agent 跳过了某一步 | 步骤不够强制性 | 加"禁止跳过""必须执行" |
| Agent 输出格式不对 | 没有指定输出格式 | 加输出模板 |
| Skill 之间冲突 | 触发条件重叠 | 细化每个 Skill 的触发词 |
